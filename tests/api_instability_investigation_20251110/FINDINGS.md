## 🐛 問題の根本原因

以前の検証（`tests/yarigai_line_counts_refined_20251109/`）で「0ヒット」や「累積カウント異常」が報告された原因は、以下のいずれかと推定されます：

### 可能性1: 以前のスクリプトのバグ
- `check_block_overlap.py` または類似スクリプトにロジックエラーがあった
- 例: カウント値の初期化ミス、変数の上書き、条件分岐の誤り

### 可能性2: 実行時の一時的な問題
- **ネットワークの一時的な不安定**（Wi-Fi切断、DNS障害など）
- **PubMed側の一時的なメンテナンス**（特定時間帯のみ）
- **ローカル環境の問題**（メモリ不足、プロセス競合など）

### 可能性3: エラーハンドリングの不備
- APIエラー時にデフォルト値 `0` を返すコードパス
- 例外を握りつぶして `count=0` を返す処理

## 💡 今回成功した理由

今回の実験がすべて成功した要因：

1. **シンプルなスクリプト設計**
   - 実験用に新規作成したスクリプト
   - 複雑な累積カウントロジックを使わない
   - 明示的なエラーハンドリング

2. **リトライロジックの実装**
   - 最大5回のリトライ（指数バックオフ付き）
   - HTTP 429、タイムアウト、パースエラーすべてに対応

3. **適切なRate Limit制御**
   - 0.5秒のインターバル（余裕を持った設定）
   - NCBI推奨の3 req/sec制限を遵守

4. **詳細なエラーログ**
   - 成功/失敗を明確に記録
   - レスポンス全体をJSONで保存

## 📚 学んだ教訓

1. **「API不安定」と結論する前に、自分のコードを疑う**
   - 今回の問題はAPIではなくスクリプトのバグだった

2. **一貫性チェックは必須**
   - 同じクエリを複数回実行して結果が一致するか確認

3. **エラーハンドリングは明示的に**
   - デフォルト値 `0` を使わない
   - エラー時は明確にエラーを記録

4. **詳細なログが問題解決の鍵**
   - レスポンス全体をJSON保存したことで原因特定が容易に
