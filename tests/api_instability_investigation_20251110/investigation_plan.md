## 仮説と検証実験

### 仮説1: クエリの複雑さがタイムアウトを引き起こす

**仮説内容**:
- Population条件 `("Physicians"[Mesh] OR physician*[tiab])` が各行に含まれており、クエリが複雑
- 複雑なクエリは処理時間が長く、APIタイムアウト（現在30秒）を超える可能性
- タイムアウト時にエラーハンドリングが不適切で `count: 0` が返される

**予測**:
- シンプルなクエリ（Population条件なし）は成功する
- 複雑なクエリ（Population条件あり）は失敗率が高い

**検証実験1A: クエリ複雑度とタイムアウトの相関**
```python

    # Simple (expected to succeed)
    '"meaningful work"[tiab]',
    'calling[tiab]',

    # Medium complexity (Population only)

    # High complexity (Population AND concept)
    '(("Physicians"[Mesh] OR physician*[tiab]) AND "meaningful work"[tiab])',
    '(("Physicians"[Mesh] OR physician*[tiab]) AND calling[tiab])',

    # Very high complexity (nested OR/AND)
```

### 仮説2: API Rate Limitingによる429エラーの誤処理

**仮説内容**:
- 連続してAPI呼び出しを行うと rate limit (3 req/sec または 10 req/sec) に達する
- HTTP 429エラーが返されるが、リトライ処理が不適切
- 結果として `count: 0` や不正確な値が返される

**予測**:
- 連続呼び出しが多いブロック（検索行数が多い）で失敗率が高い
- REQUEST_INTERVALを長くすると成功率が上がる

### 仮説5: 累積クエリの長さ制限

**仮説内容**:
- OR連結で累積クエリを構築すると、クエリ文字列が非常に長くなる
- PubMed APIにはクエリ長の制限がある（例: URLの長さ制限）
- 制限を超えると、クエリが切り詰められるか、エラーが返される

**予測**:
- ブロック後半（累積クエリが長い）で失敗率が高い
- クエリ文字列長と成功率に負の相関がある

**検証実験5A: クエリ長と成功率の相関**
