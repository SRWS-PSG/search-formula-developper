## 🐛 バグの詳細分析

### `check_block_overlap.py` の問題箇所

**ファイル**: `scripts/search/term_validator/check_block_overlap.py`
**問題行**: 220行目

```python
cumulative_count = cumulative_result['count'] if cumulative_result['count'] is not None else previous_cumulative
```

**問題点**:
1. 累積検索が失敗すると `count` が `None` になる
2. その場合、**前の累積値をそのまま使う**
3. 結果として累積カウントが増えない、または減少する

**実例** (#2B Meaningful Work):

| Line | 個別 | 累積（報告） | 期待値 | 状況 |
|------|------|------------|--------|------|
| 1 | 0→50 | 50 | 50 | ✓ 正常 |
| 2 | 0→3 | **0** | 53 | ❌ エラー（前の値50が失われた） |
| 3 | 11 | **0** | 64 | ❌ エラー（Line 2の0を引き継いだ） |
| 4 | 0→42 | 99 | 106 | ⚠️ 部分的に回復 |
| 5 | 0→6 | **0** | 112 | ❌ エラー |

**根本原因**:
- API呼び出しは成功しているが、**累積クエリの構築**に問題があった可能性
- または、**一時的なネットワークエラー**で特定の累積検索だけ失敗

## 💡 なぜ `five_year_counts_2021plus.md` は正確だったのか

### 使用されたスクリプトの推測

`five_year_counts_2021plus.md` は以下のいずれかの方法で作成されたと推測：

1. **個別クエリを1つずつ手動または別スクリプトで実行**
   - 累積カウントロジックを使わなかった
   - 各検索語を独立して実行

2. **異なるバージョンの `check_block_overlap.py` を使用**
   - バグ修正前のバージョン
   - またはカスタマイズされたバージョン

3. **PubMed Web UIでの手動確認**
   - 最も確実な方法
   - 時間はかかるが誤りがない

### 証拠

`five_year_counts_2021plus.md` のヘッダーを確認：

```markdown
# やりがい検索式: 2021年以降の件数一覧

| 行番号 | 検索式 | PubMed総件数 | 2021年以降 | 備考 |
```

- 「累積カウント」の列がない
- 各検索語が**独立して**記録されている
- これが正確性の理由

## 🔧 推奨される対策

### 即座に実施すべきこと

#### 1. `check_block_overlap.py` のバグ修正

**優先度**: 🔴 **Critical**

**修正箇所**: 行220

**現在のコード**:
```python
cumulative_count = cumulative_result['count'] if cumulative_result['count'] is not None else previous_cumulative
```

**修正案**:
```python
if not cumulative_result.get('success') or cumulative_result['count'] is None:
    # エラー時は累積カウントを明示的に記録し、処理を継続しない
    print(f"  [ERROR] 累積検索に失敗: {cumulative_result.get('message', 'Unknown error')}")
    # オプション1: デフォルト値を使わず、明示的にエラーとして記録
    cumulative_count = None
    # オプション2: 個別カウントを足し合わせて推定
    # cumulative_count = sum(r['individual_count'] for r in results) + individual_count
else:
    cumulative_count = cumulative_result['count']
```

**追加の改善**:
- リトライロジックの追加（今回の実験スクリプトと同様）
- 詳細なエラーログ（レスポンス全体をJSON保存）
- 一貫性チェック（同じクエリを複数回実行）

## 📚 教訓

### 1. データの一貫性チェックは必須

- 複数のデータソースがある場合、必ず比較する
- 不一致があれば、どちらが正しいか検証する
- 今回は `five_year_counts_2021plus.md` が正確だったが、最初に気づかなかった

### 2. 「0ヒット」を疑う

- 医学文献検索で主要な概念が0ヒットは稀
- 特に "job satisfaction" のような一般的な用語が0ヒットは異常
- 0ヒットが多い場合、スクリプトのバグを疑うべき

### 3. エラーハンドリングは明示的に

- デフォルト値 `0` や `previous_value` を使うと、エラーが隠蔽される
- エラー時は明確にエラーとして記録し、処理を停止または警告を出す
- 「なんとなく動いている」コードは危険

### 4. 一貫性チェックの重要性

- 同じクエリを複数回実行して結果が一致するか確認
- APIが不安定と結論する前に、自分のコードを疑う

### 5. 詳細なログが問題解決の鍵

- 今回の実験スクリプトでレスポンス全体をJSON保存したことで、問題がないことが明確になった
- 本番スクリプトでも詳細なログを残すべき
